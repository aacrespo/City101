Roles: Skills, Commands, and Agents

Three mechanisms in Claude Code for automating work. Here's what each one is technically, and why we use each for different purposes.

The three mechanisms

ConceptWhat it isWhere it livesHow triggeredContext
Slash Command A markdown prompt template .claude/commands/name.md You type /name Runs inline (your conversation)
Skill A richer prompt with YAML metadata .claude/skills/name/SKILL.md You type /name or Claude auto-triggers Inline or isolated (configurable)
Agent Definition An isolated worker with its own permissions .claude/agents/name.md Claude delegates automatically Isolated context window (separate from you)

What we use for our roles

/analyst, /cartographer, /modeler, /visualizer, /builder

Skill — They auto-trigger when Claude recognizes the task type. They restrict which tools each role uses. They support argument hints for quick invocation.

/session-start, /session-end, /verify-data

Skill (manual only) — These have side effects (writing LOCKBOARD, committing, pushing). Set disable-model-invocation: true so they only run when you explicitly ask.

QA verification

Agent — Runs in its own isolated context. Read-only tools. Fast model (haiku). Max 5 turns. Checks data and reports back without polluting your conversation.

What the YAML frontmatter adds

Without frontmatter, a command is just a text prompt. With it, it becomes a configurable workflow:

--- name: analyst # Display name + /slash-command description: Data collection and API work # Tells Claude WHEN to suggest it allowed-tools: Read, Grep, Bash, WebSearch # No Rhino MCP for analysts! model: inherit # Use whatever model the session uses argument-hint: [dataset or question] # Shows in autocomplete --- Read @datasets/INVENTORY.md and @LEARNINGS.md then help with: $ARGUMENTS

Git Strategy

Version control for two people, text files, and binary files. Three sub-topics: commands, branches, and binary file handling.

Native git vs GitHub CLI (gh)

TaskCommandToolWhen
Save your workgit add . && git commit -m "[DATA] add ridership"gitEvery session
Share with teammategit pushgitEnd of session
Get teammate's workgit pullgitStart of session
Create a branchgit checkout -b henna/thermalgitStart of task
Undo a mistakegit revert HEADgitWhen needed
Open a pull requestgh pr create --title "Add thermal data"ghReady to merge
Review teammate's PRgh pr view 12ghCode review
File an issuegh issue create --title "Missing Nyon data"ghTrack a problem

Rule of thumb: git = daily work (add, commit, push, pull). gh = collaboration (pull requests, issues, reviews).

Binary file handling (Git LFS)

Some files are text (CSVs, code, markdown) — git can show line-by-line diffs. Others are binary (QGIS projects, PDFs, images) — git can only track "version A → version B". Git LFS (Large File Storage) handles binaries.

Git-tracked (diffable)

*.csv — all 33 datasets
*.md — docs, context, learnings
*.html — site + visualizations
*.js — maps module, data
*.py — 8 analysis scripts
*.geojson — 6 map layers
~15 MB total

LFS-tracked (pointer only)

*.qgz — QGIS projects
*.3dm — Rhino files (future)
*.gpkg — GeoPackages
*.pdf — print maps (300 DPI)
*.png — map exports (150 DPI)
*.tif — raster data
~350 MB tracked

.gitignore (excluded)

source/ — 3 GB classmate data
output/ — staging area
archive/ — superseded files
__pycache__/
*.swp
.DS_Store
~3 GB excluded
QGIS/Rhino protocol: Binary files can't be merged. Rule: one owner at a time. Check LOCKBOARD.md before opening CITY101_WORKING.qgz. LFS tracks versions but if two people edit simultaneously, one person's work gets lost.
source/ folder (3 GB): Classmate submissions, Swisstopo GeoPackages. Too large for GitHub LFS free tier (1 GB). Solution: .gitignore and share via cloud storage (Google Drive, Polybox). These files are frozen — they don't change.

Building New Automations

How to create, iterate, and share custom commands, skills, and agents. The progression from simple to powerful.

Decision tree: what to build

I need Claude to do something repeatedly
↓ Is it a simple prompt I'll type manually?
YES → Slash Command
.claude/commands/name.md — just write the prompt as markdown. Done.
↓ Should Claude sometimes trigger it automatically?
YES → Skill
.claude/skills/name/SKILL.md — add frontmatter: description, allowed-tools, model
↓ Does it need its own isolated context?
YES → Agent Definition
.claude/agents/name.md — add frontmatter: tools, permissionMode, maxTurns, model
↓ Should it run on a schedule?
YES → Scheduled Task (Desktop) or cron + claude --print (CLI)
Desktop sidebar → Schedule. Or: crontab + claude --non-interactive

Three-step creation workflow

1
Start simple: plain command
Write a .md file in .claude/commands/ with just the prompt text. No YAML needed. Test with /command-name. Iterate the prompt until it works.
.claude/commands/cuisine-check.md
WHY: Get the prompt right before adding complexity
2
Promote to skill
Move to .claude/skills/name/SKILL.md. Add YAML frontmatter for allowed-tools, description, argument-hint. Now Claude can auto-suggest it and tools are restricted.
.claude/skills/cuisine-check/SKILL.md
WHY: Tool restriction prevents accidents. Auto-suggest saves typing.
3
Extract agent (if needed)
Copy to .claude/agents/name.md. Add permissionMode, maxTurns, model. Now it runs as an isolated subagent — parallel-safe, context-clean.
.claude/agents/cuisine-researcher.md
WHY: Isolated context = can spawn 3 of them in parallel without collision
Sharing: Everything in .claude/ is git-tracked. When Henna does git pull, she gets all the same commands, skills, and agents automatically. No installation needed.

Benefits & Trade-offs

What we gain, what changes, and what stays exactly the same.

What we gain

📜

Version history

Every change recorded. Revert any commit. See who did what.

🤝

Real collaboration

git pull / push. No more "did you see my handoff?"

🎯

Role-based context

Each role loads only what it needs. Faster, more focused agents.

📋

Automatic coordination

LOCKBOARD.md updated by /session-end. No manual task tracking.

↩️

Safe to experiment

git revert = undo any mistake. Branches = try things without risk.

🤖

Parallel agents

Spawn 3 subagents for data collection. 3x faster research sprints.

Trade-offs

Git learning curve

Andrea and Henna need to learn basic git (pull, push, branch, commit). Claude Code handles most of it through /session-end, but understanding the concepts helps.

Binary files can't merge

QGIS projects (.qgz) are binary. If two people edit at once, one version wins. Solution: LOCKBOARD.md ownership protocol.

source/ too large for GitHub

3 GB of classmate data exceeds LFS free tier. Solution: .gitignore and share via cloud storage. These files are frozen anyway.

Setup time

~3 hours to set up the repo and migrate. But the time is front-loaded — every session after that is faster.

What doesn't change

Same Claude brain. Claude Code uses the exact same model. The conversation feels the same.

Same MCP tools. Rhino, QGIS, Chrome, OSM — all available in Claude Code.

Same project. Same datasets, same visualizations, same design system. Just better organized.
